**Interfaces — B-022 (2026-07-23), gateway tool credential encryption:** `POST /v1/gateway/tools` now actually persists the `credentials` object it was always documented (`api/openapi.yaml`'s `ToolCreate.credentials`) but never implemented — previously silently discarded (decoded, never read, never written to `gateway_tools.credentials_encrypted`, 201 returned anyway; a gap surfaced by a prior full-application audit, not previously tracked under its own B-ID). New package `eami-api/internal/toolcreds` (`toolcreds.go`) implements AES-256-GCM encrypt/decrypt with a hex-encoded 32-byte key (`TOOL_CREDENTIALS_ENCRYPTION_KEY` env var, `config.Config.ToolCredentialsEncryptionKey`) — encryption happens in the API process; Postgres never sees the key or the plaintext. Deliberately not pgcrypto (the column's own schema comment says "encrypted using pgcrypto," but that extension is otherwise unused anywhere in the codebase — using it would mean sending the key to Postgres as a query parameter on every write, a weaker posture than app-side AEAD). `Server` gets a `toolCreds *toolcreds.Cipher` field (nil if the key is unconfigured — optional at startup, same convention as the B-002 gateway proxy config) and a `toolStoreOverride`/`toolQueries()` test seam scoped to `tools.go` only (mirrors `storeIface`/`WithGatewayClient` elsewhere in this file, doesn't touch the broader `Store` interface). `CreateTool` decodes `credentials` as `json.RawMessage` and a `credentialsProvided` helper decides presence structurally (any JSON object with ≥1 key, regardless of key names) rather than via the typed `ToolCredentials` struct — this was a security-review-caught bug in an earlier version of this change: deciding presence via the typed struct meant any unrecognized field name (a typo, a future field) decoded to an all-empty struct and silently reproduced the exact bug being fixed. The raw submitted bytes are what gets encrypted and stored, not a re-marshaled struct, so nothing the client sends can be dropped by the struct's fixed field set. If credentials are submitted but no key is configured, the request fails closed (500, no store call, no tool row created) — never a silent-success-with-data-loss. Credentials are write-only: `store.GatewayTool`/`ToolResp` have no field for the encrypted blob, and no query in `tools.sql.go` selects `credentials_encrypted` back out, so there is no leak surface in `ListTools`/`CreateTool`'s responses (structurally, not just by convention).

**Interfaces — B-023 (2026-07-24), real `TestTool` connectivity check:** `POST /v1/gateway/tools/{toolId}/test` no longer returns a synthetic "connected" result — it was a stub (`tools.go`'s old comment: "For now returns a synthetic 'connected' result after a brief probe" — there was no probe). New file `eami-api/internal/api/tool_connectivity.go` performs a real check per `type`: `rest_api` does a real `http.Client` GET to `base_url` (`Authorization: Bearer <api_key>` if present, HTTP Basic using `oauth_client_id`/`oauth_client_secret` as a best-effort fallback for non-standard "OAuth" APIs that accept it, `auth_type=basic` always reports `misconfigured` since `ToolCredentials` has no username/password fields to supply it) — 401/403 classify as `auth-failed`, any other completed response (including 4xx/5xx) as `connected` (reaching the server and getting *a* response proves DNS/TLS/network work, which is what the test is actually asserting). `database` does a real `pgx.ConnectConfig` handshake against `connection_string` — a successful connect already proves both reachability and valid credentials (Postgres's own auth happens during the handshake), and a failure is classified via SQLSTATE: class `28xxx` (`invalid_password`/`invalid_authorization_specification`) → `auth-failed`, anything else → `unreachable`. `mcp` always reports `misconfigured` with an explanatory message: `mcp_command` describes a local subprocess meant to run on the gateway host, not eami-api's cloud process — executing an admin-supplied command string from the SaaS backend would itself be a command-injection/RCE surface, and `eami-gateway/internal/proxy` (read for this task) confirmed the gateway's real dispatch path doesn't even consult `gateway_tools` per-tool either, so there's no existing per-tool MCP connection behavior to mirror. Every outcome is bounded by an 8-second context timeout (`toolTestTimeout`) applied once, wrapping both the decrypt step and the dial — no retries, per this task's scope. Response shape changed from the old stub's `{status, latency_ms}` to `{success, latency_ms, error}`, which is what `api/openapi.yaml` has documented for this route all along (another previously-undetected spec-vs-implementation mismatch, now closed). The richer 4-way outcome (`connected`/`auth-failed`/`unreachable`/`misconfigured`) is only in the HTTP response — `gateway_tools.status`'s DB `CHECK` constraint only allows 3 values (`connected`/`degraded`/`disconnected`, schema frozen, not touched by this task), so `toolTestOutcome.dbStatus()` maps `auth-failed` → `degraded` (reached the endpoint; credentials are what's wrong) and both `unreachable`/`misconfigured` → `disconnected` for the persisted column only.

**Security-critical addition, found by review, not in the original plan:** `eami-api` is EAMI's own cloud SaaS backend (per `ARCHITECTURE.md`), architecturally distinct from `eami-gateway`, which runs on-prem inside the customer's own network. Without a network-target guard, an org-scoped `admin`/`operator` (a role with no other legitimate reach into EAMI's cloud environment) could point a tool's `base_url`/`connection_string` at cloud metadata endpoints (`169.254.169.254`) or internal-only services and use the connected/auth-failed/unreachable distinction as a reachability oracle against EAMI's own infrastructure — a real boundary crossing this feature would otherwise introduce, not a restatement of capability the caller already has via `eami-gateway` (which runs *in* the customer's own network, where they already have legitimate reach). Added `safeDialContext` + `isBlockedTestTarget`: resolves the target host (or parses it directly if already an IP literal), rejects the request before dialing if *any* resolved address is loopback, link-local (covers `169.254.169.254` and `fe80::/10`), unspecified, or private/RFC1918/ULA (via Go's own `net.IP.IsLoopback`/`IsPrivate`/`IsLinkLocalUnicast`/`IsLinkLocalMulticast`/`IsUnspecified`), then dials the already-validated IP directly (not the original host) so a DNS answer that changes between the check and the actual connection (rebinding) can't slip through. Wired into both paths: `testRESTTool` via `http.Transport.DialContext`, `testDatabaseTool` via `pgx.ParseConfig` + `cfg.DialFunc` + `pgx.ConnectConfig` (replacing the simpler `pgx.Connect(ctx, connString)`, which has no dial-level control point). Security review re-verified the fix clean, tracing the `169.254.169.254`/RFC1918 cases end-to-end through the real (non-test) code path and confirming the new `Server.toolDialOverride` test-injection seam (needed so tests can exercise real round-trips against local `httptest` servers, which the guard would otherwise correctly reject) is assigned only in `_test.go` files — never reachable from any HTTP request.

**Other fixes from the general code-review pass, applied before commit:** (a) `safeDialContext` originally only dialed the first resolved address; fixed to fall back through every validated address in order, mirroring `net.Dialer`'s own multi-address behavior — a host whose first A/AAAA record is down but whose second works no longer incorrectly reports `unreachable`. (b) `testRESTTool`'s per-call `http.Transport` had keep-alives enabled with nothing ever calling `CloseIdleConnections()` — since the ad-hoc Transport is discarded after each call but the underlying connection and its `readLoop`/`writeLoop` goroutines stay reachable through the open connection itself, every `TestTool` call against a REST tool leaked one idle socket/goroutine; fixed with `DisableKeepAlives: true` (a one-shot connectivity check has no reuse benefit anyway). (c) `testDatabaseTool`'s deferred `conn.Close(context.Background())` had no deadline of its own — if the peer went unresponsive after a successful handshake, `Close` could hang past the 8s bound the rest of the design promises; fixed with a dedicated 2-second close-only context, independent of the (possibly already-expired) outer context.

**Credentials never leak, first production `Decrypt` caller:** a new `toolStore` interface method, `GetToolForTest`, is the only place `credentials_encrypted` is ever read back — implemented via `toolStoreWithConnectivity`, a thin wrapper running one raw SQL query through `s.queries.DB()` (the same escape hatch `finops.go` already uses for queries with no sqlc-style wrapper), so `eami-api/internal/store/tools.sql.go` needed zero changes (explicitly confirmed acceptable with the user before implementing, since it's a deviation from this codebase's usual sqlc-style convention). The decrypted plaintext only ever reaches an `Authorization` header or a Postgres connection string — `classifyHTTPError`/`classifyPgConnectError` return only static, categorical strings (`"timed out"`, `"connection failed"`, `"database rejected the credentials"`), never `err.Error()` on the underlying error, which could otherwise echo back request/DSN details. `toolcreds.Decrypt` failures (wrong/rotated key, corrupted blob) are a clean `misconfigured` result, never a panic, never exposing the underlying decrypt error text.

**Interfaces — B-025 (2026-07-24), config validation closes a live auth-bypass:** `internal/config/config.go`'s `Load()` **had no `validate()` at all** — `defaults()` hardcoded `ServiceKey: "changeme"` and a DSN with a `changeme` password, so an unset `API_SERVICE_KEY` env var meant `middleware.go`'s `requireServiceKey` (already correctly `subtle.ConstantTimeCompare`, untouched by this fix) would accept a literal `X-Service-Key: changeme` header from any caller against the collector write paths (`/v1/reports`, `/v1/ingest/batch`, `/v1/internal/token-usage`) — a real, working authentication bypass on a naive deployment, not a cosmetic placeholder. Added `validate(cfg *Config) error`, called at the end of `Load()` (its error already propagates to `main.go`'s existing `log.Fatalf`, so no `main.go` change was needed): rejects `cfg.ServiceKey` if empty or a known placeholder, and rejects `cfg.Database.DSN` if empty or its password segment is empty/placeholder. `defaults()` no longer ships any literal `changeme` — the default `ServiceKey`/`Database.DSN` fields are simply absent now, so an unconfigured deployment fails `validate()` instead of silently working. Two new unexported helpers, duplicated verbatim in `eami-gateway`'s config.go (separate Go module via `go.work`, no shared package introduced for ~30 lines): `isPlaceholderSecret(v string) bool` (trims + lowercases, checks against `{"", "changeme", "devpassword"}`) and `dsnHasPlaceholderPassword(dsn string) bool`, which — after a security-review-caught bug in the first version (see below) — extracts the DSN's password segment via `dsnPassword()` (`scheme://user:password@host` shape, last `:`/`@` delimiters) and runs it through `isPlaceholderSecret` rather than raw substring matching.
**Security-review-caught bug, fixed before commit:** the first version of `dsnHasPlaceholderPassword` did a raw `strings.Contains(lower, ":changeme@")`-style substring check directly against the whole DSN, with no trimming. `isPlaceholderSecret` (used for `ServiceKey`) already trimmed whitespace, but this sibling function didn't — so `API_DB_PASSWORD=" changeme"` (leading space) or `API_DB_PASSWORD="changeme\r"` (a CRLF-corrupted `.env` value — this repo has already hit a real CRLF bug once, see B-020) would build a DSN like `...: changeme@host...` or `...:changeme\r@host...`, and the un-trimmed substring check would miss it, silently starting with an effectively-default password. Fixed by extracting the password segment first and validating it through the same trimmed/lowercased `isPlaceholderSecret` path as every other secret, closing the inconsistency rather than patching the substring list.

**Interfaces — B-026 (2026-07-24), persistent RSA key for user-session JWTs:** `internal/auth/auth.go`'s RS256 signing key **regenerated in memory on every container restart in the actual running system** — not because no persistence mechanism existed in the code (`NewService` already had a `loadPrivateKey` path for a non-empty `keyPath`), but because `internal/config/config.go` had **no env-var override for `cfg.Auth.RSAPrivateKeyPath` at all**, and neither `eami-api.yaml` (ships `rsa_private_key_path: ""`) nor `docker-compose.yml`/`.prod.yml` set one — so the docker-deployed system was always in ephemeral dev-mode, force-logging-out every user on every restart. Mirrors `eami-gateway/internal/identity`'s `loadOrGenerateKey` pattern (read as the reference implementation, not modified by this task): new `loadOrGenerateKey(path string) (*rsa.PrivateKey, error)` in `auth.go` tries the existing `loadPrivateKey` (unchanged — still supports both PKCS1 `"RSA PRIVATE KEY"` and PKCS8 `"PRIVATE KEY"` PEM formats, a bit more flexible than gateway's PKCS1-only loader), and only if the error is specifically `os.ErrNotExist` (checked via `errors.Is`, correctly unwraps through `loadPrivateKey`'s `%w`-wrapped `fmt.Errorf`) generates a new 2048-bit key, `os.MkdirAll`s the parent dir (`0700`), and writes PKCS1-PEM at `0600`.
